Provide an analysis of lastHeatTouchedAt, lastTouchedAt, and Recency (as used in heat score and how it's related to last heat touched and last touched). I'd like to simplify this system.


Spent time tracing these fields through the schema, scoring engine, API routes, and UI components so we can see exactly how they interact before proposing simplifications. Here’s what I found.

Field Roles

lastHeatTouchedAt lives in the schema purely for the manual adjustment subsystem (lib/db/schema.ts (lines 52-70)). Only heat/cool mutations touch it (app/api/tasks/[id]/heat/route.ts (lines 108-129), app/api/tasks/[id]/cool/route.ts (lines 111-132)) plus the “touch-all” backfill for brand-new tasks (app/api/tasks/touch-all/route.ts (lines 10-55)). The scoring engine uses it exclusively to determine asymmetric decay of the stored adjustment (lib/scoring/heat-v3.ts (lines 360-405)), so general edits do not freeze or refresh decay.
lastTouchedAt records the most recent any interaction. Heat/cool also update it, but so do star toggles (app/api/tasks/[id]/star/route.ts (lines 52-77)), the generic touch endpoint (app/api/tasks/[id]/touch/route.ts (lines 10-61)), and optimistic UI handlers (lib/queries/use-task-mutations.ts:588-647, 709-733, 815-833). The UI relies on it for “new task” state (components/tasks/task-row.tsx (line 43)) and for the tooltip copy in the heat badge (components/tasks/heat-badge.tsx (lines 338-356)).
Recency in the heat score is a tiny additive component: exponential decay with a 7‑day half-life, capped at 5 points per HEAT_CONFIG.RECENCY_POINTS (lib/scoring/heat-config.ts (lines 14-36)). calculateHeat multiplies calculateRecency(lastTouchedAt) by those 5 points before summing with importance and the decayed adjustment (lib/scoring/heat-v3.ts (lines 171-240)). Because heat/cool set lastTouchedAt, any manual adjustment also maxes recency; pure edits run through /touch to get the same bump.
Observed Couplings

Heat/cool actions reset both timestamps simultaneously, so every manual adjustment gives you 5 recency points and also restarts decay. General edits call /touch, which recomputes heat solely to refresh the recency component (lib/queries/use-mark-task-touched.ts block at lib/queries/use-task-mutations.ts (lines 588-647)). This is a lot of machinery for a 5-point bonus.
touch-all exists just to clear the “new” styling without inflating recency (it copies createdAt into both fields so the recency math still reflects task age rather than “never touched”) (app/api/tasks/touch-all/route.ts (lines 10-55)).
resolveAdjustmentForTargetHeat assumes that once the user clicks heat/cool we can treat both timestamps as “now” when computing the target (lib/scoring/heat-v3.ts (lines 559-603)), which means those actions always refresh both components even if the user only wanted to jiggle manual ordering.
Simplification Paths

Drop the Recency term entirely. Remove the recency component from calculateHeat/calculateHeatWithBreakdown and the associated “touch” mutation logic. lastTouchedAt can then be limited to UI concerns (new-badge, “last touched” strings), and heat calculations depend only on importance plus the decaying adjustment. This eliminates /api/tasks/[id]/touch, the optimistic touch handler, and the need to recompute heat for non-heat edits.
Gate Recency to meaningful actions. If you still want a freshness bump, base it on max(lastHeatTouchedAt, lastMeaningfulInteractionAt) where “meaningful” is a much smaller set (heat/cool, drag/drop, maybe completion). That lets you stop updating lastTouchedAt on every star/field edit (app/api/tasks/[id]/star/route.ts (lines 52-77), general PATCH handlers) and could even let you retire /touch. Implementation would mean either (a) reuse lastHeatTouchedAt for both decay and recency or (b) add a single lastInteractionAt column and derive recency from it while keeping lastHeatTouchedAt for decay.
Quantize recency instead of timestamp math. Another simplifier is to store an interactionBucket or boolean isUntouched and compute the 0/5 points bucketed on the fly, so you don’t need to recalc heat every time lastTouchedAt changes. For example, cap recency at discrete tiers (untouched, <7 days, ≥7 days) and compute them from createdAt + lastHeatTouchedAt. That keeps the user-facing “new” behavior but removes the need for precise timestamps in scoring.
Any of these would shrink the number of moving parts: fewer simultaneous timestamp updates, less double-bookkeeping, and less chance of the race conditions called out in docs/bug_docs/heat-jumping-race-condition-analysis.md. If you want to pursue one, next steps would be to decide which behaviors (newness badge, freshness bump, manual adjustment decay) are essential, then adjust lib/scoring/heat-v3.ts, the API routes, and the React mutations accordingly.
